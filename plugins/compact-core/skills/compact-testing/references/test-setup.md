# Test Setup for Compact Contracts

Reference for setting up a Compact contract project with Vitest unit tests. For the Simulator class pattern, see `references/simulator-pattern.md`. For assertions and testing patterns, see `references/assertions-and-patterns.md`.

## Project Structure

A standard Compact project with tests follows this directory layout:

```
contract/
  vitest.config.ts                  # Vitest configuration
  package.json                      # Dependencies and scripts
  tsconfig.json                     # TypeScript configuration
  src/
    <name>.compact                  # Compact source code
    index.ts                        # Re-exports generated code + witnesses
    witnesses.ts                    # Witness implementations + private state type
    managed/                        # Generated by compact compile (do not edit)
      <name>/contract/
        index.js                    # Generated Contract class, Ledger type, ledger()
        index.d.ts                  # TypeScript declarations for generated code
    test/
      <name>-simulator.ts          # Simulator class wrapping contract operations
      <name>.test.ts               # Vitest test file
```

The `managed/` directory is created by the Compact compiler and should not be edited by hand. It is typically listed in `.gitignore` because it can be regenerated from the `.compact` source. The test files live alongside the source in `src/test/` and import from both the generated code and the witness implementations.

### Key Files

| File | Purpose |
|------|---------|
| `<name>.compact` | Compact smart contract source |
| `index.ts` | Re-exports generated code and witnesses as a single import point |
| `witnesses.ts` | Private state type definition and witness function implementations |
| `managed/<name>/contract/index.js` | Compiler-generated JavaScript implementation of the contract |
| `managed/<name>/contract/index.d.ts` | TypeScript declarations for the generated contract (Contract class, Ledger type, ledger function) |
| `<name>-simulator.ts` | Simulator class that wraps contract lifecycle and circuit calls for testing |
| `<name>.test.ts` | Vitest test file containing test suites |

### The index.ts Re-Export Pattern

The `index.ts` file provides a single import point for both the generated contract code and witness implementations:

```typescript
// contract/src/index.ts
export * as MyContract from "./managed/mycontract/contract/index.js";
export * from "./witnesses";
```

This pattern is used by consuming applications (CLI tools, UIs) to import everything from one package. Test files typically import directly from the generated code and witnesses instead of going through `index.ts`.

## Required Dependencies

Install these packages in the `contract/` directory:

| Package | Purpose | Required |
|---------|---------|----------|
| `vitest` | Test framework for running unit tests | Yes |
| `@midnight-ntwrk/compact-runtime` | Provides `createConstructorContext`, `createCircuitContext`, `sampleContractAddress`, `CircuitContext`, `WitnessContext` | Yes |
| `@midnight-ntwrk/midnight-js-network-id` | Provides `setNetworkId("undeployed")` for test environments | Yes |
| `typescript` | TypeScript compiler for type checking | Yes |

### Installation

```bash
npm install --save-dev vitest typescript
npm install @midnight-ntwrk/compact-runtime @midnight-ntwrk/midnight-js-network-id
```

The `compact-runtime` and `midnight-js-network-id` packages are runtime dependencies (not just dev dependencies) because they are also used in production code (witnesses, DApp integration). The Compact compiler itself (`compact`) is installed separately via the Midnight toolchain and is not an npm dependency.

### Version Compatibility

The version of `@midnight-ntwrk/compact-runtime` must match the version expected by the Compact compiler that generated the `managed/` output. A version mismatch causes runtime errors when instantiating the contract. If you update the compiler, also update `compact-runtime` to the corresponding version.

## Vitest Configuration

Create a `vitest.config.ts` in the `contract/` directory:

```typescript
import { defineConfig } from "vitest/config";

export default defineConfig({
  mode: "node",
  test: {
    deps: { interopDefault: true },
    globals: true,
    environment: "node",
    include: ["**/*.test.ts"],
    exclude: ["node_modules"],
    root: ".",
  },
  resolve: {
    extensions: [".ts", ".js"],
    conditions: ["import", "node", "default"],
  },
});
```

### Configuration Options Explained

| Option | Value | Why It Matters |
|--------|-------|----------------|
| `mode` | `"node"` | Ensures the Node.js environment is used, which is required for cryptographic operations in `compact-runtime` (field arithmetic, hashing) |
| `deps.interopDefault` | `true` | **Critical.** The Compact compiler generates CommonJS modules (`index.cjs`). This setting enables proper default export interop when importing CommonJS modules from ESM test files. Without it, imports from the generated code fail silently or return `undefined`. |
| `globals` | `true` | Makes `describe`, `it`, `expect`, `beforeAll`, `beforeEach` available globally without explicit imports. Reduces boilerplate. |
| `environment` | `"node"` | Runs tests in a Node.js environment rather than jsdom or other browser-like environments |
| `include` | `["**/*.test.ts"]` | Matches test files by the `.test.ts` suffix convention |
| `root` | `"."` | Sets the test root to the `contract/` directory |
| `resolve.extensions` | `[".ts", ".js"]` | Allows importing both TypeScript source and compiled JavaScript files |
| `resolve.conditions` | `["import", "node", "default"]` | Controls how the module resolver picks between ESM and CJS exports in `package.json` `exports` fields. This order ensures ESM imports are preferred, falling back to Node.js-specific and default conditions. Required for the mixed module ecosystem of Midnight packages. |

### Optional: Coverage and Reporters

The official examples include coverage and CI reporting configuration:

```typescript
export default defineConfig({
  // ... base config above ...
  test: {
    // ... base test config ...
    coverage: {
      provider: "v8",
      reporter: ["text", "json", "html"],
    },
    reporters: ["default", ["junit", { outputFile: "reports/report.xml" }]],
  },
});
```

The `v8` coverage provider works out of the box with no additional dependencies. The `junit` reporter generates XML reports suitable for CI systems like GitHub Actions.

## Network ID Setup

Before any contract operations in tests, you must call `setNetworkId("undeployed")`:

```typescript
import { setNetworkId } from "@midnight-ntwrk/midnight-js-network-id";

setNetworkId("undeployed");
```

### Why setNetworkId Is Required

The Midnight runtime uses a network identifier to configure address formats, transaction serialization, and other network-specific behavior. In production, this is set to the target network (`"testnet"`, `"devnet"`, etc.). In tests, `"undeployed"` tells the runtime that there is no real blockchain -- this is a local simulation environment.

### Where to Call It

Three common placement patterns (all are idempotent -- calling multiple times with the same value has no effect):

**Module-level (recommended, used in official examples):** Executes once when the module loads, before any test runs.

```typescript
import { setNetworkId } from "@midnight-ntwrk/midnight-js-network-id";

setNetworkId("undeployed");

describe("My contract", () => { /* tests */ });
```

**`beforeAll` hook:** Useful when you want explicit lifecycle control or share setup across multiple test suites.

```typescript
import { setNetworkId } from "@midnight-ntwrk/midnight-js-network-id";

beforeAll(() => {
  setNetworkId("undeployed");
});
```

**Inside the Simulator constructor:** Ensures the network is configured whenever a simulator is created, co-locating setup with the object that needs it.

```typescript
export class MySimulator {
  constructor() {
    setNetworkId("undeployed");
    // ... contract setup
  }
}
```

### What Happens Without It

If you skip `setNetworkId`, contract instantiation or circuit execution throws a network configuration error. The error message varies but typically indicates that the network ID has not been set. This is one of the most common setup mistakes.

## package.json Scripts

Standard scripts for a Compact contract project with tests:

```json
{
  "scripts": {
    "compact": "compact compile src/<name>.compact src/managed/<name>",
    "build": "npm run compact && tsc",
    "test": "vitest run",
    "test:compile": "npm run compact && vitest run"
  }
}
```

### Script Reference

| Script | Command | Purpose |
|--------|---------|---------|
| `compact` | `compact compile src/<name>.compact src/managed/<name>` | Compiles the `.compact` source into the `managed/` output directory. Generates `index.js`, `index.d.ts`, ZK keys, and ZKIR files. |
| `build` | `npm run compact && tsc` | Compiles the Compact source, then runs the TypeScript compiler to produce distributable JavaScript. Used for publishing the contract package. |
| `test` | `vitest run` | Runs all Vitest tests once (no watch mode). Assumes the Compact source has already been compiled (the `managed/` directory exists). |
| `test:compile` | `npm run compact && vitest run` | Compiles the Compact source first, then runs tests. Use this for full validation -- if the contract fails to compile, the tests never run. |

### When to Use Each Script

- **During development:** Use `npm run test` after an initial `npm run compact`. You only need to recompile when you change the `.compact` file, not when you change test files or witnesses.
- **Before committing:** Use `npm run test:compile` to verify everything from scratch.
- **In CI pipelines:** Use `npm run test:compile` or separate the steps (`npm run compact && npm test`) so compilation failures are clearly distinguished from test failures.

### The compact compile Command

```bash
compact compile <input>.compact <output-dir>
```

- **Input:** A `.compact` source file (e.g., `src/counter.compact`)
- **Output directory:** Where the generated files are placed (e.g., `src/managed/counter`)

The output directory structure:

```
src/managed/<name>/
  compiler/            # Compiler metadata
  contract/
    index.js           # Generated JavaScript contract implementation
    index.js.map       # Source map back to the .compact file
    index.d.ts         # TypeScript declarations
  keys/                # ZK proving and verifying keys
  zkir/                # ZK intermediate representation
```

Exit code 0 indicates successful compilation. Any non-zero exit code means the contract has errors -- see `references/compilation-validation.md` for common compiler errors and how to interpret them.

## TypeScript Configuration

A minimal `tsconfig.json` for the contract directory:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "declaration": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  },
  "include": ["src/**/*.ts"],
  "exclude": ["node_modules", "dist", "src/test"]
}
```

Key settings:

| Option | Value | Why |
|--------|-------|-----|
| `module` | `"Node16"` | Required for proper ESM/CJS interop with the generated CommonJS modules |
| `moduleResolution` | `"Node16"` | Matches the module setting for correct resolution of `.js` imports |
| `esModuleInterop` | `true` | Enables default import interop with CommonJS modules |
| `skipLibCheck` | `true` | Skips type checking of `.d.ts` files in `node_modules`, which speeds up compilation |

The `exclude` array should include `src/test` for the build configuration (`tsconfig.build.json`) so test files are not included in the published package. A separate `tsconfig.json` without this exclusion is used for type checking during development.

## Minimal Working Example

Bringing all the pieces together, here is the minimum setup to run a single test:

**1. Install dependencies:**

```bash
npm install @midnight-ntwrk/compact-runtime @midnight-ntwrk/midnight-js-network-id
npm install --save-dev vitest typescript
```

**2. Compile the contract:**

```bash
compact compile src/counter.compact src/managed/counter
```

**3. Create `vitest.config.ts`** (as shown above)

**4. Write a test file** (`src/test/counter.test.ts`):

```typescript
import { setNetworkId } from "@midnight-ntwrk/midnight-js-network-id";
import {
  createConstructorContext,
  createCircuitContext,
  sampleContractAddress,
} from "@midnight-ntwrk/compact-runtime";
import { Contract, ledger } from "../managed/counter/contract/index.js";

setNetworkId("undeployed");

describe("Counter", () => {
  it("should start at zero", () => {
    const contract = new Contract({});
    const { currentPrivateState, currentContractState, currentZswapLocalState } =
      contract.initialState(
        // Second arg is an entropy seed for ZK proofs â€” a 64-char hex string.
        // All-zeros is safe for local tests; use real randomness in production.
        createConstructorContext({}, "0".repeat(64)),
      );
    const circuitContext = createCircuitContext(
      sampleContractAddress(),
      currentZswapLocalState,
      currentContractState,
      currentPrivateState,
    );
    const state = ledger(circuitContext.currentQueryContext.state);
    expect(state.counter).toEqual(0n);
  });
});
```

**5. Run the test:**

```bash
npx vitest run
```

This is the bare minimum. For real projects, use the Simulator pattern (see `references/simulator-pattern.md`) to avoid repeating the context setup boilerplate in every test.

## Common Setup Mistakes

| Mistake | Symptom | Fix |
|---------|---------|-----|
| Missing `setNetworkId("undeployed")` | Network configuration error on contract instantiation | Add `setNetworkId("undeployed")` before any contract operations |
| Missing `deps.interopDefault: true` in vitest config | Imports from generated code return `undefined` or fail | Add `deps: { interopDefault: true }` to the `test` config |
| Wrong import path for generated code | Module not found error | Use `../managed/<name>/contract/index.js` (include the contract subdirectory) |
| `compact-runtime` version mismatch | Runtime errors during contract instantiation | Ensure `compact-runtime` version matches the compiler version |
| Running tests without compiling first | Import errors (managed directory missing) | Run `npm run compact` before `npm test`, or use `npm run test:compile` |
| Using `.cjs` extension in imports | Resolution failures in ESM context | Use `.js` extension; the resolver handles CJS interop via `interopDefault` |
| Missing `resolve.conditions` in vitest config | Module resolution failures with Midnight packages | Add `conditions: ["import", "node", "default"]` to `resolve` config |
