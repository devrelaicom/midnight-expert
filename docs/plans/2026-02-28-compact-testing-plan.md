**Goal:** Create a `compact-testing` skill in the compact-core plugin that teaches how to unit test Compact smart contracts using the compiled JavaScript implementation, Vitest, and the Simulator pattern, plus compilation validation.

**Architecture:** The skill follows the same structure as existing compact-core skills: a SKILL.md entry point with frontmatter trigger description, a references/ directory with deep-dive docs, and an examples/ directory with complete annotated code files. Content is sourced from Midnight's official documentation, example repos (counter, bboard), and the compact-runtime API.

**Tech Stack:** Markdown (skill docs), TypeScript (example test files), Vitest (test framework), `@midnight-ntwrk/compact-runtime` (test infrastructure)

## Important Context

### Plugin Location
All files go in: `plugins/compact-core/skills/compact-testing/`

### Existing Skill Pattern
Every skill in this plugin follows this structure:
- `SKILL.md` with YAML frontmatter (`name`, `description`) — description is the trigger
- `references/*.md` — deep reference docs, no frontmatter
- `examples/*` — complete annotated code files

### Key Technical Details (from MCP research)
The Compact compiler generates a JavaScript implementation (`index.cjs`) that can be tested directly. Key imports from `@midnight-ntwrk/compact-runtime`:
- `createConstructorContext(initialPrivateState, seed)` — creates constructor context
- `createCircuitContext(contractAddress, zswapState, contractState, privateState)` — creates circuit context
- `sampleContractAddress()` — generates a dummy contract address for tests
- `WitnessContext<Ledger, PrivateState>` — type for witness function first argument
- `CircuitContext` — type for the mutable test context

From generated code (`managed/*/contract/index.js`):
- `Contract<PS>` — main contract class, generic over private state
- `Ledger` — typed ledger state interface
- `ledger()` — extracts typed Ledger from raw state data

Also required: `setNetworkId("undeployed")` from `@midnight-ntwrk/midnight-js-network-id`

Circuit calls: `contract.impureCircuits.<name>(circuitContext, ...args)` returns `{ context, result }`
Ledger reads: `ledger(circuitContext.currentQueryContext.state)`
Private state: `circuitContext.currentPrivateState`
Error testing: `.toThrow("failed assert: <message>")`

   ```

       index.ts                    # Re-exports + witnesses

         <name>.test.ts            # Vitest test file
   ```

2. **Required Dependencies** — Table of packages and their purpose:
   - `vitest` — test framework
   - `@midnight-ntwrk/compact-runtime` — runtime for testing (createConstructorContext, createCircuitContext, sampleContractAddress, WitnessContext, CircuitContext)
   - `@midnight-ntwrk/midnight-js-network-id` — required for setNetworkId("undeployed")

3. **Vitest Configuration** — Complete `vitest.config.ts`:
   ```typescript
   import { defineConfig } from "vitest/config";

   export default defineConfig({
     mode: "node",
     test: {
       deps: { interopDefault: true },
       globals: true,
       environment: "node",
       include: ["**/*.test.ts"],
       exclude: ["node_modules"],
       root: ".",
     },
     resolve: {
       extensions: [".ts", ".js"],
       conditions: ["import", "node", "default"],
     },
   });
   ```
   - `deps.interopDefault: true` is critical — the generated `index.cjs` is CommonJS and needs interop
   - `mode: "node"` ensures Node.js environment for crypto operations
   - `conditions: ["import", "node", "default"]` resolves ESM/CJS properly

4. **Network ID Setup** — Explain `setNetworkId("undeployed")`:
   - Must be called before any contract operations
   - Tells the runtime this is a test environment with no real blockchain
   - Typically placed in a `beforeAll` or at the top of the test file
   - Without it, contract instantiation throws a network configuration error

5. **package.json Scripts**:
   ```json
   {
     "scripts": {
       "compact": "compact compile src/<name>.compact src/managed/<name>",
       "build": "npm run compact && tsc",
       "test": "vitest run",
       "test:compile": "npm run compact && vitest run"
     }
   }
   ```
   - `test` runs tests only (assumes already compiled)
   - `test:compile` compiles then tests (full validation)

## Task 2: Create references/simulator-pattern.md

**Files:**
- Create: `plugins/compact-core/skills/compact-testing/references/simulator-pattern.md`

**Step 1: Create the reference file**

Write `references/simulator-pattern.md` covering:

1. **Why Use a Simulator** — The Simulator class is the recommended pattern from Midnight's official examples. It encapsulates:
   - Contract instantiation with witnesses
   - Constructor context creation and initial state
   - Circuit context lifecycle (updating after each call)
   - Convenience methods for calling circuits
   - Ledger and private state accessors

2. **Anatomy of a Simulator** — Complete annotated example:
   ```typescript
   import {
     type CircuitContext,
     createConstructorContext,
     createCircuitContext,
     sampleContractAddress,
     WitnessContext,
   } from "@midnight-ntwrk/compact-runtime";
   import { setNetworkId } from "@midnight-ntwrk/midnight-js-network-id";
   import {
     Contract,
     type Ledger,
     ledger,
   } from "../managed/<name>/contract/index.js";

   // 1. Define private state type
   type MyPrivateState = {
     readonly secretKey: Uint8Array;
   };

   // 2. Define witness implementations
   const witnesses = {
     local_secret_key: ({
       privateState,
     }: WitnessContext<Ledger, MyPrivateState>): [MyPrivateState, Uint8Array] =>
       [privateState, privateState.secretKey],
   };

   // 3. Build the simulator class
   export class MySimulator {
     private contract: Contract<MyPrivateState>;
     private circuitContext: CircuitContext<MyPrivateState>;

     constructor(secretKey: Uint8Array) {
       setNetworkId("undeployed");

       this.contract = new Contract<MyPrivateState>(witnesses);

       // Run the constructor circuit to initialize state
       const { currentPrivateState, currentContractState, currentZswapLocalState } =
         this.contract.initialState(
           createConstructorContext({ secretKey }, "0".repeat(64)),
         );

       // Create the circuit context for subsequent calls
       this.circuitContext = createCircuitContext(
         sampleContractAddress(),
         currentZswapLocalState,
         currentContractState,
         currentPrivateState,
       );
     }

     // 4. Wrap each circuit as a method
     doSomething(arg: bigint): void {
       this.circuitContext = this.contract.impureCircuits.doSomething(
         this.circuitContext,
         arg,
       ).context;
     }

     // 5. Expose ledger state
     getLedgerState(): Ledger {
       return ledger(this.circuitContext.currentQueryContext.state);
     }

     // 6. Expose private state
     getPrivateState(): MyPrivateState {
       return this.circuitContext.currentPrivateState;
     }
   }
   ```

3. **Context Lifecycle** — Explain the flow:
   - `createConstructorContext(privateState, seed)` creates the initial context for the contract constructor
   - `contract.initialState(constructorContext)` runs the constructor and returns `{ currentPrivateState, currentContractState, currentZswapLocalState }`
   - `createCircuitContext(address, zswapState, contractState, privateState)` creates the reusable circuit context
   - After each circuit call, `result.context` contains the updated context — you MUST reassign it
   - The seed (`"0".repeat(64)`) is a 64-character hex string used for deterministic key derivation in tests

4. **Calling Circuits** — Two categories:
   - **Impure circuits** (state-changing): `contract.impureCircuits.<name>(ctx, ...args)` — returns `{ context, result }`
   - **Pure circuits** (no state): `contract.circuits.<name>(ctx, ...args)` — returns `{ context, result }` but context is unchanged
   - **Pure circuit helpers**: `contract.pureCircuits.<name>(...args)` — returns the result directly, no context needed

5. **Witness Function Signature** — The pattern every witness follows:
   ```typescript
   (witnessContext: WitnessContext<Ledger, PrivateState>, ...args): [PrivateState, ReturnValue]
   ```
   - First param is always `WitnessContext<Ledger, PrivateState>` containing `privateState` and ledger access
   - Returns a tuple: `[updatedPrivateState, returnValue]`
   - To mutate private state, return a new object: `[{ ...privateState, field: newValue }, returnValue]`
   - Object keys must exactly match Compact witness function names

6. **Multi-User Testing** — Switch the private state to simulate different users:
   ```typescript
   switchUser(secretKey: Uint8Array): void {
     this.circuitContext.currentPrivateState = { secretKey };
   }
   ```

7. **Type Mapping Quick Reference** — Essential mappings for test values:
   | Compact | TypeScript Test Value | Example |
   |---|---|---|
   | `Field` | `bigint` | `0n`, `42n` |
   | `Uint<64>` | `bigint` | `100n` |
   | `Bytes<32>` | `Uint8Array` | `new Uint8Array(32)` |
   | `Boolean` | `boolean` | `true`, `false` |
   | `Opaque<"string">` | `string` | `"hello"` |
   | `Maybe<T>` | `{ is_some: boolean; value: T }` | `{ is_some: true, value: "hi" }` |
   | `enum` variants | `number` | `State.ACTIVE` (0), `State.INACTIVE` (1) |
   | `Counter` (ledger) | `bigint` | via `ledger().counter` |

   ```typescript
   expect(privateState.secretKey).toEqual(expectedKey);
   ```

4. **Error Case Testing** — Compact `assert` statements throw with the assertion message:
   ```typescript
   // Test that a circuit rejects invalid input
   expect(() =>
     simulator.doSomething(invalidInput),
   ).toThrow("failed assert: Only owner can perform this action");

   // Test specific error messages
   expect(() =>
     simulator.post("message"),
   ).toThrow("failed assert: Attempted to post to an occupied board");
   ```

5. **Testing Constructors** — Verify the initial state after deployment:
   ```typescript
   const simulator = new MySimulator(secretKey);
   const state = simulator.getLedgerState();

   // Constructor should have initialized these
   expect(state.phase).toEqual(Phase.EMPTY);
   expect(state.owner).toEqual(expectedPublicKey);
   expect(state.round).toEqual(0n);
   ```

6. **Multi-User / Multi-Actor Testing**:
   ```typescript
   import { randomBytes } from "crypto";

   // User A deploys and posts
   const userA = randomBytes(32);
   const simulator = new BBoardSimulator(userA);
   simulator.post("Hello from A");

   // User B tries to take down A's post — should fail
   const userB = randomBytes(32);
   simulator.switchUser(userB);
   expect(() => simulator.takeDown()).toThrow(
     "failed assert: Attempted to take down post, but not the current owner",
   );

   // Switch back to A — should succeed
   simulator.switchUser(userA);
   simulator.takeDown(); // no error
   ```

7. **Pure Circuit Testing** — Testing computational helpers:
   ```typescript
   // Pure circuits run without state context
   const result = contract.pureCircuits.computeHash(inputBytes);
   expect(result).toBeInstanceOf(Uint8Array);
   expect(result.length).toEqual(32);

   // Verify determinism
   const result2 = contract.pureCircuits.computeHash(inputBytes);
   expect(result).toEqual(result2);
   ```

8. **Sequential Operations** — Testing state transitions:
   ```typescript
   it("should track state across multiple operations", () => {
     const sim = new CounterSimulator();
     expect(sim.getLedgerState().round).toEqual(0n);

     sim.increment();
     expect(sim.getLedgerState().round).toEqual(1n);

     sim.increment();
     expect(sim.getLedgerState().round).toEqual(2n);

     sim.decrement();
     expect(sim.getLedgerState().round).toEqual(1n);
   });
   ```

9. **Edge Cases** — What to test at boundaries:
   - Default/zero values after construction
   - Maximum `Uint` values (`BigInt("18446744073709551615")` for `Uint<64>`)
   - Empty `Bytes<N>` (`new Uint8Array(N)`)
   - `Maybe` with `is_some: false`
   - Operations on empty `Map`, `Set`, `Counter`

## Task 4: Create references/compilation-validation.md

**Files:**
- Create: `plugins/compact-core/skills/compact-testing/references/compilation-validation.md`

**Step 1: Create the reference file**

Write `references/compilation-validation.md` covering:

1. **Compilation as a Test Gate** — The `compact compile` command is the first line of defense. It catches:
   - Type errors (wrong types passed to circuits, incorrect return types)
   - Disclosure violations (missing `disclose()` wrappers)
   - Sealed ledger misuse (writing to sealed fields outside constructor)
   - Syntax errors (wrong pragma format, incorrect enum syntax)
   - Undefined references (using undeclared variables or types)

2. **The compile Command**:
   ```bash
   compact compile src/<name>.compact src/managed/<name>
   ```
   Input: a `.compact` source file
   Output directory structure:
   ```
   src/managed/<name>/
   ├── compiler/        # Compiler metadata
   ├── contract/
   │   ├── index.js     # Generated JavaScript implementation
   │   ├── index.js.map # Source map back to .compact
   │   └── index.d.ts   # TypeScript declarations
   ├── keys/            # ZK proving/verifying keys
   └── zkir/            # ZK intermediate representation
   ```
   Exit code 0 = success, non-zero = failure

3. **Common Compiler Errors** — Table of errors with explanations and fixes:
   | Error Pattern | Cause | Fix |
   |---|---|---|
   | `potential witness-value disclosure must be declared` | Missing `disclose()` at a public boundary | Add `disclose()` where the witness-derived value crosses ledger/conditional/return |
   | `parse error` with expected tokens | Syntax error (wrong keyword, missing semicolon) | Check pragma format, enum dot notation, return type `[]` not `Void` |
   | `type mismatch: expected X, got Y` | Wrong type in assignment, argument, or return | Check type mapping; use `as` for casts (Uint -> Field -> Bytes) |
   | `sealed ledger field cannot be assigned outside constructor` | Writing to a `sealed` field in a circuit | Only assign sealed fields in the constructor |
   | `undefined reference` | Using a variable, type, or function not declared | Check imports, ensure types are exported, verify spelling |
   | `Version mismatch` at runtime | `compact-runtime` version doesn't match compiler | Update `@midnight-ntwrk/compact-runtime` to match compiler version |

4. **Integrating into CI** — Scripts for CI pipelines:
   ```yaml
   # GitHub Actions example
   - name: Compile Compact contracts
     run: npx compact compile src/contract.compact src/managed/contract

   - name: Run unit tests
     run: npx vitest run
   ```
   Best practice: compile first, then test. The `test:compile` script combines both.

5. **Pragma Version Compatibility**:
   ```compact
   pragma language_version >= 0.16 && <= 0.18;
   ```
   - Must use bounded ranges without patch versions
   - The pragma defines which compiler versions can compile this contract
   - Mismatch causes a compiler error, not a runtime error

6. **Using the MCP Compiler** — The Midnight MCP server provides `midnight-compile-contract` for quick validation without a local compiler install. Options:
   - `skipZk=true` (default): Fast syntax-only validation (~1-2s)
   - `fullCompile=true`: Full compilation with ZK circuit generation (~10-30s)
   - Check `validationType` in response: `"compiler"` = real compilation, `"static-analysis-fallback"` = fallback

```typescript
// ============================================================================
// VITEST CONFIGURATION FOR COMPACT CONTRACT TESTING
// ============================================================================
//
// This configuration is optimized for testing Compact smart contracts that
// compile to CommonJS JavaScript modules (index.cjs). The key settings are:
//
//   - deps.interopDefault: true — Required for importing CommonJS modules
//     generated by the Compact compiler. Without this, default exports
//     from index.cjs will not resolve correctly.
//
//   - mode: "node" — Ensures Node.js environment for crypto operations
//     used by @midnight-ntwrk/compact-runtime (field arithmetic, hashing).
//
//   - resolve.conditions — Ensures ESM/CJS module resolution works
//     correctly with the mixed module ecosystem of Midnight packages.
//
// Usage:
//   npx vitest run          # Run all tests once
//   npx vitest              # Run in watch mode
//   npx vitest run --reporter=verbose  # Verbose output
//
// ============================================================================

import { defineConfig } from "vitest/config";

export default defineConfig({
  mode: "node",
  test: {
    // interopDefault is CRITICAL for Compact — the compiler generates
    // CommonJS modules (index.cjs) and this setting ensures proper
    // default export handling when importing them in ESM test files.
    deps: { interopDefault: true },

    // Enable global test functions (describe, it, expect) without
    // explicit imports. Reduces boilerplate in test files.
    globals: true,

    // Use Node.js environment for crypto and buffer operations
    // required by the compact-runtime library.
    environment: "node",

    // Standard test file patterns
    include: ["**/*.test.ts"],
    exclude: ["node_modules"],
    root: ".",
  },
  resolve: {
    // Support both TypeScript and JavaScript imports
    extensions: [".ts", ".js"],

    // Module resolution conditions for the Midnight package ecosystem
    conditions: ["import", "node", "default"],
  },
});
```

No official test framework — you test `index.cjs` directly. This skill covers the Simulator pattern, context lifecycle, assertions, and compilation validation. Cross-reference to `compact-witness-ts` for witness implementation details and `compact-structure` for contract anatomy.

2. **Quick Start** — Minimal annotated code to get a test running (10-15 lines: imports, setNetworkId, Contract, initialState, createCircuitContext, circuit call, ledger assertion).

3. **Simulator Pattern** — Summary paragraph + skeleton of the Simulator class (constructor, circuit methods, ledger/private accessors). Reference `references/simulator-pattern.md` for full details.

4. **Testing Checklist** — Table:
   | What to Test | How | Pattern |
   |---|---|---|
   | Circuit produces correct ledger state | `ledger(ctx.currentQueryContext.state).field` | Ledger assertion |
   | Circuit returns correct value | `.result` from circuit call | Return assertion |
   | Circuit rejects bad input | `expect(() => ...).toThrow("failed assert: ...")` | Error testing |
   | Private state updates correctly | `ctx.currentPrivateState.field` | Private state assertion |
   | Multi-user interaction | Switch `currentPrivateState` between calls | Multi-actor |
   | Pure computation | `contract.pureCircuits.<name>(args)` | Pure circuit test |
   | Contract compiles | `compact compile` exits 0 | Compilation gate |

5. **Key API Reference** — Table:
   | Function / Type | Import From | Purpose |
   |---|---|---|
   | `Contract<PS>` | Generated code | Main contract class |
   | `Ledger` | Generated code | Typed ledger interface |
   | `ledger()` | Generated code | Extract typed ledger from state |
   | `createConstructorContext()` | `@midnight-ntwrk/compact-runtime` | Create constructor context |
   | `createCircuitContext()` | `@midnight-ntwrk/compact-runtime` | Create circuit execution context |
   | `sampleContractAddress()` | `@midnight-ntwrk/compact-runtime` | Generate dummy address for tests |
   | `CircuitContext<PS>` | `@midnight-ntwrk/compact-runtime` | Type for mutable test context |
   | `WitnessContext<L, PS>` | `@midnight-ntwrk/compact-runtime` | Type for witness first argument |
   | `setNetworkId()` | `@midnight-ntwrk/midnight-js-network-id` | Required test setup |

6. **Compilation Validation** — Short paragraph: `compact compile` is the first test gate. It catches type errors, disclosure violations, sealed ledger misuse. Reference `references/compilation-validation.md`.

7. **Common Mistakes** — Table:
   | Mistake | Fix |
   |---|---|
   | Forgetting `setNetworkId("undeployed")` | Call it before any contract operations |
   | Not updating context after circuit call | Always reassign: `ctx = result.context` |
   | Using `contract.circuits` for impure ops | Use `contract.impureCircuits` for state-changing circuits |
   | Expecting string equality on `Bytes<N>` | Compare `Uint8Array` with `.toEqual()`, not `===` |
   | Missing `deps.interopDefault: true` in vitest config | Required for CommonJS `index.cjs` imports |
   | Testing with wrong `compact-runtime` version | Must match the version the compiler expects |
   | Witness key mismatch | Witness object keys must exactly match Compact `witness` names |

8. **Reference Routing** — Two tables:
   - References table (topic → file)
   - Examples table (example → file → pattern)
